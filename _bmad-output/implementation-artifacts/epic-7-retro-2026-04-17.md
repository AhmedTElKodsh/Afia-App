## Challenges & Growth Areas

1. **Epic planning gap — 3 stories missing from original backlog**
   Stories 7.6, 7.7, 7.8 were in the PRD (FR44, FR46, FR47) but not converted to stories during epic planning. They were added and implemented same-day via Course Correction. The work was not hard — it was simply unplanned.

2. **TF.js / Canvas API mocking in Node.js test environment**
   Story 7.4 has 4/20 failing tests; Story 7.8 has 6/15 failing tests. Root cause: TF.js tensor chain mocking (`toFloat`, `expandDims`, `div`) and Canvas API are not available in jsdom. Discovered mid-implementation, not anticipated upfront.

3. **Story status drift — 3 of 8 story files show wrong status**
   Stories 7.4 ("review"), 7.6 ("ready-for-dev"), 7.7 ("in-progress") are marked done in sprint-status.yaml but their own files reflect intermediate states. Erodes document trust.

4. **Version Management UI deferred (Story 7.5, Task 5)**
   Admin cannot activate/rollback model versions from the admin panel — requires direct Supabase access. Acceptable for controlled testing; becomes an operational risk when Stage 2 activates at scale.

## Key Insights

- **PRD completeness ≠ epic completeness.** Every FR in PRD scope needs an explicit story before sprint starts.
- **The 500-scan gate is the real Stage 2 trigger** — not a calendar date. Training data is already ahead of plan.
- **Stage 1 prompt quality gates everything.** If LLM estimates are noisy during data collection, Stage 2 trains on bad ground truth.
- **FR28 (camera orientation guide) is a data quality gate, not UX polish.** Inconsistent bottle orientation in training images degrades model accuracy.
